# devlog 핸드오프 템플릿

> 사용자가 "핸드오프"라고 하면 Opus는 이 템플릿에 맞춰 이번 세션 내용을 출력한다.
> 사용자는 출력된 텍스트를 C:\dev\devlog.md 맨 위에 붙여넣는다.

## 헤더 포맷

```
## YYYY-MM-DD HH:mm KST | 프로젝트명 | 사건명 | 키워드1,키워드2,키워드3
```

최신 항목이 파일 맨 위. 헤더의 키워드가 검색 인덱스 역할을 한다.

## 섹션 구성

### 목표
[사용자 중심 · 문장형]
이 세션을 시작한 배경과 달성하려 한 것을 2~3문장으로 쓴다.
6개월 뒤 맥락 없이 읽어도 "왜 이 세션이 필요했는지" 알 수 있어야 한다.

### 조사 내용
[AI 중심 · 리스트형 · 조사가 있었을 때만 작성]
- 조사 주제 — 핵심 결론 (출처)

### 시도 순서
[AI 중심 · 번호 리스트형]
1. [시도 내용] → [결과: 성공/실패] → [실패 시 원인 또는 성공 시 맥락]

### 선택과 기각
[AI 중심 · 리스트형]
- 채택: [무엇] — 이유: [왜] — 맥락: [누가/어떤 상황에서 이 방향을 결정했는지]
- 기각: [무엇] — 이유: [왜] — 맥락: [누가/어떤 상황에서 이 방향을 꺾었는지]

### 변경 파일
[AI 중심 · 테이블형 · 코드 변경이 있었을 때만 작성]
| 파일 경로 | 변경 내용 |
|---|---|
| `경로` | 내용 |

코드 변경이 없었으면: (이 세션에서 코드 변경 없음)

### 이 세션에서 해결 못한 것
[사용자 중심 · 문장형]
- OPEN: [해결 안 된 것을 문장으로]
- ASSUMED: [확인 없이 가정한 것을 문장으로]

### 검증
- [ ] 목표에 이 세션을 시작한 배경이 포함되었는가
- [ ] 시도 순서에 실패 포함
- [ ] 기각 사유에 맥락 포함
- [ ] OPEN 항목 표기 완료
- [ ] 조사 내용에 출처 포함
- [ ] AI 참조 섹션(조사 내용, 시도 순서, 선택과 기각, 변경 파일)이 생략 없이 작성되었는가

## 작성 규칙

- 사용자 중심 섹션(목표, 이 세션에서 해결 못한 것): 자연스러운 문장형. 사용자가 직접 읽는다.
- AI 중심 섹션(조사 내용, 시도 순서, 선택과 기각, 변경 파일): 구조화된 리스트/테이블형. Opus가 문제 추적 시 참조한다. 사용자가 읽지 않더라도 절대 생략하지 않는다.
- 검증 체크리스트: 템플릿 맨 끝 고정 (Lost in the Middle 대응).
- 각 항목 사이에 --- 구분선을 넣지 않는다. 핸드오프 끝에만 --- 하나를 넣어 다음 항목과 구분한다.

## CHANGELOG와의 차이

| | CHANGELOG | devlog |
|---|---|---|
| 기록 | 뭘 고쳤는가 (결과) | 왜, 어떻게, 뭘 시도했는가 (과정) |
| 읽는 시점 | 배포·업데이트 확인 | 사고·문제 원인 추적 |
| 포맷 | 버전 기준 | 시간 기준 |

## 기존 인수인계서와의 관계

인수인계서(common-rules.md "세션 인수인계")와 devlog 핸드오프는 별개다.
- 인수인계서: 다음 세션을 바로 이어가기 위한 문서. 트리거: "마무리/세션 종료/인수인계/handover"
- devlog 핸드오프: 영구 기록용 개발 일지. 트리거: "핸드오프"
- 동시 요청 시 둘 다 출력한다.
