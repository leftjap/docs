# WORKFLOW.md — 오늘의네비 피드백지침서 관리 가이드

## 이 문서의 용도

이 문서는 AI가 오늘의네비 피드백지침서의 수정 요청을 받았을 때 따라야 하는 규칙이다.

**관리 대상 파일:**
- `C:\dev\docs\글쓰기\오늘의네비_피드백지침서.md`

**작업 흐름 요약:**
1. 사용자가 이 문서(WORKFLOW.md)를 업로드하고 수정 요청을 보낸다.
   (피드백지침서 파일이 함께 업로드되면 그것을 우선 참조한다)
2. AI는 이 문서를 읽고 요청을 분석한다.
3. AI는 섹션 상세 맵(5번)에서 관련 섹션을 특정한 뒤,
   업로드된 파일이 없으면 GitHub raw URL로 직접 크롤링한다.
4. 트랙 A이면 바로 작업지시서를 출력한다.
   트랙 B이면 방향 확인서를 먼저 출력한다.
5. 사용자가 작업지시서를 VS Code 에이전트에 복사해서 실행한다.
6. VS Code 에이전트는 문서 수정 → git add → git commit → git push를 모두 완료한다.

---

## 0. 작업 흐름

### 프로토콜

AI는 사용자의 요청을 받으면, 먼저 **트랙 A(즉시 진행)** 또는 **트랙 B(방향 확인)** 중 어느 쪽인지 판단한다.

---

#### 트랙 A — 즉시 진행

**조건 (모두 충족해야 한다):**
- 요청이 명확하다 (어느 섹션의 무엇을 어떻게 바꿀지 특정할 수 있다)
- 기존 섹션 내 문장 수정/추가이다 (새 섹션 추가가 아니다)
- 문서의 톤이나 역할 정의가 변하지 않는다

**예시:** "하지 않는 것"에 항목 추가, 레퍼런스 톤 예시 추가, 문장 수정

**흐름:**
1. 사용자가 이 문서(WORKFLOW.md)와 수정 요청을 보낸다.
   대상 문서 파일이 함께 업로드되면 그것을 우선 참조한다.
2. AI는 요청을 분석하고, 필요하면 GitHub raw URL에서 대상 문서를 크롤링한다.
3. AI는 **바로 작업지시서를 출력한다.**

---

#### 트랙 B — 방향 확인 후 진행

**조건 (하나라도 해당하면 트랙 B):**
- 새 섹션을 추가한다
- 기존 섹션의 역할이나 톤 정의를 변경한다
- "너는 누구인가"의 정체성 규정이 바뀐다
- 요청이 모호하거나 해석이 여러 가지다
- 기존 규칙과 모순되는 내용을 추가한다

**흐름:**
1. 사용자가 이 문서(WORKFLOW.md)와 수정 요청을 보낸다.
2. AI는 요청을 분석하고 **방향 확인서**를 출력한다.
3. 사용자가 승인하면 AI는 **바로 작업지시서를 출력한다.** 재확인하지 않는다.
4. 사용자가 방향을 수정하면, AI는 수정된 방향으로 방향 확인서를 다시 출력한다.

---

#### 트랙 판단 규칙

- AI는 매 요청마다 트랙을 판단하고, 트랙 A이면 작업지시서 상단에 `[트랙 A]`를, 트랙 B이면 방향 확인서 상단에 `[트랙 B]`를 표기한다.
- 판단이 애매하면 트랙 B를 선택한다.
- 사용자가 "바로 만들어", "작업지시서 바로 줘" 등 즉시 진행을 명시하면, 트랙 B 조건이더라도 트랙 A로 전환한다.

---

### 방향 확인서 형식 (트랙 B에서만 사용)

```
방향 확인: [요청 요약]

요청 이해
[사용자의 요청을 AI가 어떻게 이해했는지 1~3문장으로 정리]

변경 내용
[어느 섹션의 무엇을 어떻게 바꿀 것인지]

기존 규칙과의 충돌 여부
[새 내용이 기존 "하지 않는 것" 또는 "너는 누구인가"와 모순되지 않는지]

대안 (있을 경우)
[다른 접근법이 있으면 제시]
```

## 1. 작업지시서 출력 규칙

### 형식

```
⚠️ 모든 Step을 빠짐없이 순서대로 실행하세요. 특히 마지막 커밋 Step을 절대 생략하지 마세요.

프로젝트 경로 (모든 Step에서 이 절대 경로를 사용하세요)
프로젝트: C:\dev\docs\

작업지시서: [변경 요약]
작업 유형: [내용 추가 / 내용 수정 / 구조 변경]

Step 1
파일: C:\dev\docs\글쓰기\오늘의네비_피드백지침서.md
위치: [섹션 제목]
작업: [정확히 무엇을 추가/수정/삭제하는지]
교체 내용: [수정할 블록 전체 — 복사-붙여넣기만으로 완료 가능해야 한다]
완료 확인: [이 단계가 끝나면 어떤 상태여야 하는지]

WORKFLOW.md 갱신 (해당 시)
5번 섹션 상세 맵: [추가/삭제/변경된 섹션]

Step N — 커밋 & 푸시
명령어:
  cd "C:\dev\docs"
  git add -A
  git commit -m "[작업유형]: [변경 요약]"
  git push origin main

커밋 메시지 형식: [작업유형]: [변경 요약]
작업유형: add / update / restructure
완료 확인: git push 성공, 에러 없이 종료
```

### 내용 제공 규칙

이 문서는 산문 중심이므로 블록 단위가 크립토 문서와 다르다.

**블록의 단위:**
- 섹션 전체 (## 제목부터 다음 ## 전까지)
- 독립 문단 (빈 줄로 구분된 하나의 단락)
- 레퍼런스 톤의 예시 하나 (일기 인용 + 피드백 한 쌍)

**금지 패턴:**
- ❌ "너는 누구인가 섹션에서 세 번째 문단의 두 번째 문장을 수정한다" (위치 모호)
- ✅ "너는 누구인가 섹션의 세 번째 문단을 아래 내용으로 교체한다" (문단 전체 제공)

**새 내용 추가 시:**
- 삽입 위치를 "X 섹션의 마지막 문단 아래" 수준으로 명시한다.
- 추가할 내용 전체를 제공한다.

### WORKFLOW.md 갱신 규칙

피드백지침서 변경으로 아래가 바뀌면 WORKFLOW.md 갱신 Step을 포함한다:
- 5번(섹션 상세 맵): 섹션 추가·삭제·제목 변경, 레퍼런스 톤 예시 추가

**갱신하지 않는 경우:**
- 기존 섹션 내 문장만 수정하고 구조 변경이 없는 경우
- "하지 않는 것"에 항목을 추가하되 섹션 구조는 동일한 경우

### 커밋 & 푸시 규칙

모든 작업지시서의 마지막 Step에 커밋과 푸시를 포함한다.

**마지막 Step 형식:**
```
Step N — 커밋 & 푸시
명령어:
  cd "C:\dev\docs"
  git add -A
  git commit -m "[작업유형]: [변경 요약]"
  git push origin main

커밋 메시지 형식: [작업유형]: [변경 요약]
작업유형: add / update / restructure
완료 확인: git push 성공, 에러 없이 종료
```

## 2. 절대 하지 않는 것

- AI가 피드백지침서의 톤 정의를 자의적으로 변경하지 않는다.
- "너는 누구인가"의 정체성 규정을 사용자 지시 없이 수정하지 않는다.
- "하지 않는 것" 목록의 기존 항목을 삭제하지 않는다. 추가만 한다.
- 레퍼런스 톤 예시를 AI가 자작하지 않는다. 사용자가 제공한 실제 대화만 예시로 넣는다.

## 3. 문서 서식 규칙

### 마크다운 서식

| 요소 | 규칙 |
|---|---|
| 섹션 제목 | `##` 만 사용 (현재 문서가 `##` 단일 계층) |
| 본문 | 산문. 번호 목록, 불릿 목록 사용하지 않음 |
| 강조 | 큰따옴표로 직접 인용. 굵은체/이탤릭 최소 사용 |
| 예시 블록 | 레퍼런스 톤 섹션에서만 사용. "지오의 일기:" + "피드백:" 쌍으로 |

### 톤 일관성 규칙

피드백지침서 자체의 문체도 지침서가 정의하는 톤과 일관되어야 한다. 학술적/관료적 문체로 작성하지 않는다. 말하듯이 쓴다.

## 5. 섹션 상세 맵

오늘의네비 피드백지침서의 전체 구조다. AI는 수정 요청을 받으면 이 맵에서 관련 섹션을 먼저 특정한다.

**제목줄**
- 내용: "오늘의네비.md — 일기 피드백 지침서 v2"
- 비고: 버전 번호는 큰 구조 변경 시에만 올린다

**이게 뭔가**
- 내용: 한 줄 정의 — 지오의 일기를 읽고 나누는 대화
- 분량: 1문장

**왜 존재하는가**
- 내용: 일기의 존재 이유 — "모름"을 안고 자기를 들여다보는 행위
- 분량: 3문단
- 핵심 개념: 인식의 퇴적물, 선택 = 실시간 답변

**너는 누구인가**
- 내용: AI의 정체성 정의 — 분석자가 아닌 대화 상대
- 분량: 5문단 (가장 긴 섹션, 문서의 핵심)
- 하위 규칙들:
  - 한두세 군데만 골라 깊이 들어간다
  - 평가/점수 매기기를 하지 않는다
  - 많이 읽고 넓게 아는 사람 (심리학 구체적 연구명 OK)
  - 솔직하다 (비위 맞추지 않되 깎아내리지도 않음)
  - 지오가 모르는 맥락을 얹어준다

**레퍼런스 톤**
- 내용: 톤/깊이/구조의 실제 예시
- 형식: 설명 1문단 + 예시 1쌍 (일기 인용 + 피드백)
- 현재 예시: hot-cold empathy gap (로웬스타인)
- 비고: 예시 추가 시 이 섹션에 쌍으로 추가

**하지 않는 것**
- 내용: 금지 행동 목록 (산문 형태, 불릿 아님)
- 현재 항목 수: 10개 (문장 구조 분석 ✕, 문법 교정 ✕, 리스트 출력 ✕, 양면 제시 ✕, 편집 제안 ✕, 방향 제시 ✕, 패턴 반복 ✕, 같은 수준 반복 ✕, 논지 통합 ✕, 패턴 가두기 ✕)
- 규칙: 기존 항목 삭제 금지, 추가만 허용

**분량**
- 내용: 일기 밀도에 비례하는 분량 규칙
- 분량: 1문단
- 핵심: 짧게 쓰는 것 ≠ 빈 말
