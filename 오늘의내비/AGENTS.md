# AGENTS.md — 오늘의내비 아카이브 관리 가이드

## 이 문서의 용도

이 문서는 AI가 오늘의내비 아카이브의 수정 요청을 받았을 때 따라야 하는 규칙이다.

**관리 대상 파일:**
- `C:\dev\docs\오늘의내비\지오\오늘의 내비(2026~).md` — 현재 진행 중인 일기
- `C:\dev\docs\오늘의내비\지오\오늘의 내비_2025.md` — 2025년 일기
- `C:\dev\docs\오늘의내비\지오\오늘의 내비_2022~2025.md` — 2022~2025년 일기
- `C:\dev\docs\오늘의내비\지오\피드백지침서.md` — 지오 일기 피드백 지침서
- `C:\dev\docs\오늘의내비\소연\소연 내비.md` — 소연의 일기
- `C:\dev\docs\오늘의내비\소연\피드백지침서.md` — 소연 일기 피드백 지침서

**작업 흐름 요약:**
1. 사용자가 이 문서(AGENTS.md)를 업로드하고 수정 요청 또는 새 글을 보낸다.
   (대상 파일이 함께 업로드되면 그것을 우선 참조한다)
2. AI는 이 문서를 읽고 요청을 분석하여 트랙(A/B/C)을 판단한다.
3. AI는 파일 구조 맵(5번)에서 대상 파일을 특정한 뒤,
   업로드된 파일이 없으면 GitHub raw URL로 직접 크롤링한다.
4. 트랙 A → 바로 작업지시서를 출력한다.
   트랙 B → 방향 확인서를 먼저 출력한다.
   트랙 C → 글쓴이를 판단하고 바로 작업지시서를 출력한다.
5. 사용자가 작업지시서를 VS Code 에이전트에 복사해서 실행한다.
6. VS Code 에이전트는 문서 수정 → git add → git commit → git push를 모두 완료한다.

## 0. 작업 흐름

### 프로토콜

AI는 사용자의 요청을 받으면, 먼저 **트랙 A(즉시 진행)** 또는 **트랙 B(방향 확인)** 중 어느 쪽인지 판단한다.

---

#### 트랙 A — 즉시 진행

**조건 (모두 충족해야 한다):**
- 요청이 명확하다 (어느 파일의 어느 위치에 무엇을 추가/수정할지 특정할 수 있다)
- 기존 파일 내 내용 추가 또는 수정이다 (파일 분리/병합/구조 변경이 아니다)
- 일기의 톤이나 원문을 변경하지 않는다

**예시:** 새 일기 항목 추가, 날짜 오류 수정, 아이디어 항목 추가, 오타 수정

**흐름:**
1. 사용자가 이 문서(AGENTS.md)와 수정 요청을 보낸다.
   대상 파일이 함께 업로드되면 그것을 우선 참조한다.
2. AI는 요청을 분석하고, 필요하면 GitHub raw URL에서 대상 파일을 크롤링한다.
3. AI는 **바로 작업지시서를 출력한다.**

---

#### 트랙 B — 방향 확인 후 진행

**조건 (하나라도 해당하면 트랙 B):**
- 새 파일을 생성한다
- 기존 파일을 분리하거나 병합한다
- 폴더 구조를 변경한다
- 기존 일기 원문을 삭제하거나 대폭 수정한다
- 요청이 모호하거나 해석이 여러 가지다

**흐름:**
1. 사용자가 이 문서(AGENTS.md)와 수정 요청을 보낸다.
2. AI는 요청을 분석하고 **방향 확인서**를 출력한다.
3. 사용자가 승인하면 AI는 **바로 작업지시서를 출력한다.** 재확인하지 않는다.
4. 사용자가 방향을 수정하면, AI는 수정된 방향으로 방향 확인서를 다시 출력한다.

## 트랙 C — 소연/지오 내비 글 추가 + 피드백 댓글

### 사용자 입력
- AGENTS.md (이 문서)
- 새 글 텍스트
- "소연 글" 또는 "지오 글" 명시

### 처리 흐름

#### 소연 글인 경우
1. `C:\dev\docs\오늘의내비\소연\소연 내비.md` 를 로컬에서 열어 마지막 항목의 날짜·구분선 서식을 확인하고, 동일 서식으로 새 글을 파일 끝에 삽입한다.
2. 일기 내용에서 핵심 주제/감정을 파악하고, 서재 어구록 DB에서 관련 발췌문을 검색한다.
   - `load_quotes`를 태그 또는 키워드로 호출한다. 태그 목록은 서재 AGENTS.md 섹션 6 참조.
   - 검색 결과 중 일기 맥락에 자연스럽게 맞는 발췌문이 있으면 피드백에 녹인다.
   - 맞는 게 없으면 AI 자체 지식으로 맥락을 붙인다. 어구록 인용은 의무가 아니다.
3. 피드백 지침서에 따라 댓글 텍스트를 작성한다.
4. `load_partner_db` 로 docId를 조회한다.
5. `post_comment` 를 **1회만** 호출한다. 응답에 `cmt_` ID가 포함되면 같은 docId로 재호출하지 않는다.
6. `git add . → git commit → git push origin main`
7. 판단 과정이나 중간 로그를 터미널에 출력하지 않는다. 결과만 실행한다.

2. `git add . → git commit → git push origin main`
3. 피드백 댓글 없음. (지오 일기 피드백이 별도로 요청되면 그때 어구록 DB를 검색하여 활용한다.)

### 소연 피드백 지침서

소연 피드백의 상세 지침은 `C:\dev\docs\오늘의내비\소연\피드백지침서.md`를 참조한다. 아래는 핵심 요약이다.

#### 핵심 원칙
- 대화 상대이지 일기 분석기가 아니다. 한두세 군데만 골라서 깊이 들어간다.
- 점수를 매기거나 평가하지 않는다. 생각을 나누고, 맥락이 되는 이야기를 꺼낸다.
- 소연이 모르는 맥락을 붙여준다—관련된 이론이나 연구, 비슷한 경험을 한 사람의 이야기, 책에서 읽은 장면. 단, 용어를 던지는 게 목적이 아니라 소연의 언어로 한 박자 풀어서 설명한다.
- 소연의 편인 사람이다. 힘든 날 쓴 글에는 위로가 먼저, 여유 있는 날에는 학문적 맥락을 붙인다.
- 소연이 쓴 장면의 효과를 구체적으로 말해줘서, 소연이 자기 글의 힘을 스스로 느끼게 한다. "글 잘 쓴다"고 직접 말하지 않는다.
- 소연이 글을 못 쓴 날에 죄책감을 느낄 때, 이미 쓰고 있다는 사실을 비춰주는 것으로 충분하다.

#### AI 번역투 표현 금지
- "이 장면에서 멈췄다", "이 문장이 머릿속에 남는다", "숨이 멈추는 것 같았다" 등 한국어 일상 대화에서 거의 쓰이지 않는 AI 번역투 표현을 쓰지 않는다.
- "마음이 아팠다", "울컥했다", "뭉클했다", "웃겼다" 등 일상적인 감정 표현은 자연스럽게 쓴다.
- 기준: 소연이 카톡에서 친구한테 쓸 법한 표현이면 된다. AI 글에서만 보이는 표현이면 안 된다.

#### 트랙 판단 규칙

- 새 글 원문이 포함되어 있으면 → **트랙 C**
- 트랙 C가 아닌 경우, 트랙 A 조건을 모두 충족하면 → **트랙 A**
- 트랙 A 조건을 하나라도 충족하지 못하면 → **트랙 B**
- 판단이 애매하면 트랙 B를 선택한다.
- 사용자가 "바로 만들어", "작업지시서 바로 줘" 등 즉시 진행을 명시하면, 트랙 B 조건이더라도 트랙 A로 전환한다.
- AI는 매 요청마다 트랙을 판단하고, 작업지시서 또는 방향 확인서 상단에 `[트랙 A]` / `[트랙 B]` / `[트랙 C]`를 표기한다.

## 1. 작업지시서 출력 규칙

### 형식

```
⚠️ 모든 Step을 빠짐없이 순서대로 실행하세요. 특히 마지막 커밋 Step을 절대 생략하지 마세요.

프로젝트 경로 (모든 Step에서 이 절대 경로를 사용하세요)
프로젝트: C:\dev\docs\

작업지시서: [변경 요약]
작업 유형: [내용 추가 / 내용 수정 / 구조 변경]

Step 1
파일: [절대 경로]
위치: [날짜 또는 섹션]
작업: [정확히 무엇을 추가/수정/삭제하는지]
교체 내용: [수정할 블록 전체]
완료 확인: [이 단계가 끝나면 어떤 상태여야 하는지]

Step N — 커밋 & 푸시
명령어:
  cd "C:\dev\docs"
  git add -A
  git commit -m "[작업유형]: [변경 요약]"
  git push origin main
```

### 내용 제공 규칙

이 문서는 날짜순 일기이므로 블록 단위가 다른 문서와 다르다.

**블록의 단위:**
- 하루치 일기 전체 (날짜 표기부터 다음 날짜 표기 전까지)
- 아이디어/메모의 경우 구분선(----)으로 나뉜 하나의 항목

**금지 패턴:**
- ❌ "2026년 1월 2일 일기에서 세 번째 문단을 수정한다" (위치 모호)
- ✅ "2026-01-02 일기의 끝에 아래 문단을 추가한다" (삽입 위치 명확)

**새 일기 추가 시:**
- 삽입 위치를 "파일 맨 끝" 또는 "YYYY-MM-DD 항목 뒤"로 명시한다.
- 추가할 내용 전체를 제공한다.

### AGENTS.md 갱신 규칙

아래가 바뀌면 AGENTS.md 갱신 Step을 포함한다:
- 5번(파일 구조 맵): 파일 추가·삭제·이름 변경, 하위 폴더 변경

**갱신하지 않는 경우:**
- 기존 파일에 일기 항목을 추가하거나 오타를 수정하는 경우

## 2. 절대 하지 않는 것

- AI가 일기 원문의 문체나 표현을 교정하지 않는다. 일기는 지오의 말투 그대로 보존한다.
- AI가 일기 내용을 요약하거나 재구성하지 않는다.
- 기존 일기 항목을 삭제하지 않는다. 사용자가 명시적으로 지시한 경우만 예외.
- 날짜 순서를 임의로 재배치하지 않는다.
- 일기에 등장하는 인물의 실명을 AI가 자의적으로 변경하지 않는다.

## 3. 문서 서식 규칙

### 파일별 날짜/구분 형식

| 파일 | 날짜 형식 | 항목 구분 |
|---|---|---|
| 오늘의 내비(2026~).md | `YYYY-MM-DD` | 같은 날 내 주제 전환은 번호 또는 빈 줄 |
| 오늘의 내비_2025.md | `**[YYYY-MM-DD]{.mark}**` | `---` 구분선 |
| 오늘의 내비_2022~2025.md | 볼드 번호 + 제목 (`**1.** 제목`) | 번호순 |
| 아이디어 _ 메모 _ 단편(습작).md | 없음 | `----` 구분선, `#아이디어----제목` |

### 서식 원칙

- 새 내용 추가 시 해당 파일의 기존 형식을 따른다.
- 본문은 산문. 지오의 원문 문체를 유지한다.
- 강조(볼드, 이탤릭)는 기존 파일의 방식을 따른다.

## 4. 소스 참조 프로토콜

### GitHub 정보

| 항목 | 값 |
|---|---|
| 레포 | `https://github.com/leftjap/docs` (Private) |
| GitHub raw base | `https://raw.githubusercontent.com/leftjap/docs/main/` |
| 오늘의내비 base URL | `https://raw.githubusercontent.com/leftjap/docs/main/오늘의내비/` |

### 파일 참조 우선순위

1순위 — 사용자가 업로드한 파일
2순위 — AI가 GitHub raw URL에서 직접 크롤링
3순위 — 사용자에게 업로드 요청 (1, 2 모두 불가능할 때만)

### 크기 제한

`오늘의 내비_2022~2025.md`는 2MB 이상이므로 크롤러가 잘릴 가능성이 높다. 이 파일 수정 시에는 사용자에게 업로드를 요청하거나, 수정 위치(날짜)를 먼저 특정한 뒤 해당 부분만 작업한다.
