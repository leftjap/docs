# AGENTS.md — 크립토 투자 마스터 문서 관리 가이드

> **공통 규칙**: AI의 응답은 간결한 경어체로 작성합니다.

## 이 문서의 용도

이 문서는 AI가 크립토 투자 마스터 문서의 수정 요청을 받았을 때 따라야 하는 규칙이다.

**관리 대상 파일:**
- `C:\dev\docs\투자 전략\크립토_투자_마스터_문서.md`

**작업 흐름 요약:**
1. 사용자가 이 문서(AGENTS.md)를 업로드하고 수정 요청을 보낸다.
   (크립토 문서 파일이 함께 업로드되면 그것을 우선 참조한다)
2. AI는 이 문서를 읽고 요청을 분석한다.
3. AI는 섹션 상세 맵(5번)에서 관련 섹션을 특정한 뒤,
   업로드된 파일이 없으면 GitHub raw URL로 직접 크롤링한다.
4. 트랙 A이면 바로 작업지시서를 출력한다.
   트랙 B이면 방향 확인서를 먼저 출력한다.
5. 사용자가 작업지시서를 Claude Code에 붙여넣어 실행한다.
6. Claude Code는 문서 수정 → git add → git commit → git push를 모두 완료한다.

## 0. 작업 흐름

### 프로토콜

AI는 사용자의 요청을 받으면, 먼저 **트랙 A(즉시 진행)** 또는 **트랙 B(방향 확인)** 중 어느 쪽인지 판단한다.

---

#### 트랙 A — 즉시 진행

**조건 (모두 충족해야 한다):**
- 요청이 명확하다 (어느 섹션의 무엇을 어떻게 바꿀지 특정할 수 있다)
- 기존 섹션 내 수치/날짜/문장 수정이다 (새 섹션 추가가 아니다)
- 교차 참조 영향이 없거나 1~2곳이다

**예시:** 체크리스트 항목 수치 업데이트, 표의 날짜/가격 수정, 오타 수정, 기존 표에 행 추가

**흐름:**
1. 사용자가 이 문서(AGENTS.md)와 수정 요청을 보낸다.
   대상 문서 파일이 함께 업로드되면 그것을 우선 참조한다.
2. AI는 요청을 분석하고, 필요하면 GitHub raw URL에서 대상 문서를 크롤링한다.
3. AI는 교차 참조 맵(6번)을 확인하고 **바로 작업지시서를 출력한다.**

---

#### 트랙 B — 방향 확인 후 진행

**조건 (하나라도 해당하면 트랙 B):**
- 새 섹션 또는 새 부록을 추가한다
- 기존 섹션의 구조를 변경한다 (분리, 병합, 이동)
- 교차 참조 영향이 3곳 이상이다
- 요청이 모호하거나 해석이 여러 가지다
- 기존 분석/판단과 모순되는 내용을 추가한다

**흐름:**
1. 사용자가 이 문서(AGENTS.md)와 수정 요청을 보낸다.
2. AI는 요청을 분석하고 **방향 확인서**를 출력한다.
3. 사용자가 승인하면 AI는 **바로 작업지시서를 출력한다.** 재확인하지 않는다.
4. 사용자가 방향을 수정하면, AI는 수정된 방향으로 방향 확인서를 다시 출력한다.

---

#### 트랙 판단 규칙

- AI는 매 요청마다 트랙을 판단하고, 트랙 A이면 작업지시서 상단에 `[트랙 A]`를 표기한다.
- 트랙 B이면 방향 확인서 상단에 `[트랙 B]`를 표기한다.
- 판단이 애매하면 트랙 B를 선택한다.
- 사용자가 "바로 만들어", "작업지시서 바로 줘" 등 즉시 진행을 명시하면, 트랙 B 조건이더라도 트랙 A로 전환한다.

---

### 방향 확인서 형식 (트랙 B에서만 사용)

```
방향 확인: [요청 요약]

요청 이해
[사용자의 요청을 AI가 어떻게 이해했는지 1~3문장으로 정리]

변경 내용
[어느 섹션의 무엇을 어떻게 바꿀 것인지. 구체적 섹션 번호와 변경 개요 포함]

교차 참조 영향
[이 변경으로 함께 수정해야 하는 다른 섹션 — 6번 교차 참조 맵 기반]

대안 (있을 경우)
[다른 접근법이 있으면 제시]
```

### 내용 제공 규칙

코드 프로젝트의 "함수 전체 교체" 원칙과 동일하게, 문서에서는 **블록 단위 교체**를 원칙으로 한다.

**블록의 단위:**
- 표 전체 (헤더 + 구분선 + 모든 행)
- 인용구 블록 전체 (> 로 시작하는 연속 줄)
- 번호 목록 전체 (1. 부터 마지막 항목까지)
- 독립 문단 (빈 줄로 구분된 하나의 단락)

**금지 패턴:**
- ❌ "3.7절 표에서 5월 행의 내용을 바꾼다" (위치 모호)
- ❌ "아래 문장을 찾아서 수정한다" (검색 실패 가능)
- ✅ "3.7절 '2026년 타임라인 전망' 표를 아래 표로 통째로 교체한다" (블록 전체 제공)

**새 내용 추가 시:**
- 삽입 위치를 "X.X절 [제목]의 마지막 인용구 블록 아래" 수준으로 명시한다.
- 추가할 내용 전체를 제공한다.

### AGENTS.md 갱신 규칙

크립토 문서 변경으로 아래가 바뀌면 AGENTS.md 갱신 Step을 포함한다:
- 5번(섹션 상세 맵): 섹션/하위 항목/표 추가·삭제·이동
- 6번(교차 참조 맵): 섹션 간 연결 관계 변경
- 7번(핵심 용어): 새 용어/약어 등장

**AI 필수 체크 — 작업지시서의 문서 수정 Step을 모두 작성한 뒤, 아래를 확인한다:**
1. 섹션이 추가/삭제/이동되었는가? → 5번 갱신 Step 포함
2. 새 교차 참조가 생겼는가? → 6번 갱신 Step 포함
3. 새 용어/약어가 등장했는가? → 7번 갱신 Step 포함

**갱신하지 않는 경우:**
- 기존 섹션 내 수치/날짜만 변경하고 구조 변경이 없는 경우
- 기존 표에 행을 추가하되 교차 참조에 영향이 없는 경우

### AI 응답 규칙

- 작업 규모를 부풀리지 않는다. 한 번에 할 수 있으면 묻지 않고 한 번에 한다.
- 선택지를 나열하는 것으로 끝내지 않는다. AI의 추천을 반드시 붙이고 근거를 밝힌다.
- 확신 수준을 구분한다: "확실합니다" / "높은 확률이지만 검증 필요" / "추측입니다".

### 커밋 & 푸시 규칙

모든 작업지시서의 마지막 Step에 커밋과 푸시를 포함한다.

**마지막 Step 형식:**
```
Step N — 커밋 & 푸시
명령어:
  cd "C:\dev\docs"
  git add -A
  git commit -m "[작업유형]: [변경 요약]"
  git push origin main

커밋 메시지 형식: [작업유형]: [변경 요약]
작업유형: add / update / restructure
완료 확인: git push 성공, 에러 없이 종료
```

### playbook.md 갱신 규칙

모든 작업지시서의 커밋 & 푸시 Step 직전에 playbook.md 갱신 Step을 포함한다.

- 파일: `C:\dev\playbook\playbook.md`
- 갱신 대상: 2번 백로그 표에서 해당 작업의 상태를 변경한다.
  - 완료 시: 해당 행을 삭제하고 변경 이력(7번)에 완료 기록 추가
  - 새 이슈 발견 시: 🟡 대기에 새 행 추가 (ID는 B-XX 채번)
  - 진행 상태 변경 시: 상태/메모 컬럼 갱신
- 갱신 후 별도 커밋:
  ```
  cd "C:\dev\playbook"
  git add playbook.md
  git commit -m "update: playbook.md [변경 요약]"
  git push origin main
  ```
- playbook.md 갱신이 불필요한 경우: 기존 섹션 내 수치/날짜만 변경하는 사소한 수정

## 2. 절대 하지 않는 것

- 기존 분석의 결론을 AI가 자의적으로 변경하지 않는다. 수치 업데이트는 하되, 판단은 사용자가 지시한 대로만 반영한다.
- 섹션 번호를 임의로 재배정하지 않는다. 번호 변경이 필요하면 방향 확인서에서 먼저 제안한다.
- 부록(A, B, C, D)의 순서를 변경하지 않는다.
- 변경 이력(부록 D)의 기존 항목을 수정하지 않는다. 항상 추가만 한다.

## 3. 문서 서식 규칙

### 마크다운 서식

| 요소 | 규칙 |
|---|---|
| 섹션 제목 | `##` Part, `###` 소제목, `####` 하위 (기존 계층 유지) |
| 표 | 파이프 구분, 헤더 + 구분선 + 데이터 행 |
| 인용구 | `>` 블록. 핵심 통찰/원칙/경고에 사용 |
| 굵은 이탤릭 | `***텍스트***`. 섹션 최종 요약/판정에만 사용 |
| 굵은체 | `**텍스트**`. 강조에 사용 |
| 코드 | 사용하지 않음 (이 문서는 코드 문서가 아님) |

### 표 작성 규칙

- 컬럼 수는 기존 표와 동일하게 유지한다.
- 새 행을 추가할 때 기존 행의 서식을 따른다.
- 표를 수정할 때는 표 전체(헤더 + 구분선 + 모든 행)를 교체 내용으로 제공한다.

### 인용구 작성 규칙

- 핵심 통찰, 원칙, 경고를 인용구로 감싼다.
- 인용구 내부에서 이탤릭(`*텍스트*`)으로 작성자의 주관적 판단을 표시한다.
- 인용구 블록을 수정할 때는 블록 전체를 교체 내용으로 제공한다.

### 변경 이력(부록 D) 규칙

모든 수정 시 부록 D에 항목을 추가한다:
```
- M/DD: [변경 요약]. [어느 섹션이 영향받았는지].
```

## 5. 섹션 상세 맵

크립토 투자 마스터 문서의 전체 구조다. AI는 수정 요청을 받으면 이 맵에서 관련 섹션을 먼저 특정한다.

### Part 1. 시장 구조 & 매크로 프레임워크

**1.1 핵심 원칙: 유동성이 모든 것의 원인이다**
- 내용: 유동성 → 내러티브 → 심리 인과관계
- 형식: 인용구 블록

**1.2 AI가 바꾼 시장 구조**
- 내용: 관점 수렴 → 포지션 수렴 → 유동성 수확
- 형식: 인용구 블록

**1.3 주식 vs 코인: 난이도 차이의 본질**
- 내용: 비교표 (내재가치, 거래시간, 안전장치, 유동성 깊이, 청산 메커니즘)
- 형식: 표 1개 + 공식 2줄

**1.4 황금알을 낳는 거위 논리**
- 내용: 시장 운영자 딜레마 — "죽이지 않고 털기"
- 형식: 인용구 블록

**1.5 제도화의 양날의 검**
- 내용: ETF 도입 후 새로운 하방/상방 메커니즘
- 형식: 인용구 블록 (스트럭처드 노트, 캐리 트레이드, 외부 충격)

---

### Part 2. 유동성 분석 프레임워크

**2.1 유동성 파이프라인 — M2만 보면 틀린다**
- 내용: 4단계 파이프라인 (글로벌 M2 → Net Liquidity → 스테이블코인 → 거래소 유입)
- 형식: 표 1개 (4단계) + 인용구

**2.2 현실적으로 볼 것 3개**
- 내용: Net Liquidity, QRA, 스테이블코인+거래소 유입
- 형식: 표 1개 (우선순위 3개)
- 판정 기준: "3개의 방향이 같으면 확신도 높고, 엇갈리면 관망"

**2.3 DXY 재해석: 반비례가 아니라 스트레스 게이지**
- 내용: DXY 상태별 의미 (급등/급락/횡보)
- 형식: 표 1개 + 인용구
- 핵심: "속도"를 봐야 한다
